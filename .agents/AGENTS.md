## 🔍 2. กฎการออกแบบและวิเคราะห์เทสเคส (Test Case Analysis Rules)

เมื่อเอเจนต์ต้องวิเคราะห์ แก้ไข หรือสร้างเทสเคสใหม่ในโปรเจกต์นี้ จะต้องทำตามหลักการดังนี้:

### 2.1 จำแนกเจตนาการทดสอบ (Test Intent Classification)
ทุกครั้งที่อธิบายหรือนำเสนอเทสเคส ต้องมีการแบ่งประเภทตามหลักการทดสอบดังนี้:
*   **Happy Path:** โฟลว์ปกติที่ทุกขั้นตอนเป็นไปตามเงื่อนไขทางธุรกิจปกติและราบรื่น
*   **Unhappy Path:** โฟลว์ที่มีการเบี่ยงเบนหรือเคสซับซ้อนตามเงื่อนไขธุรกิจโรงพยาบาล แต่ระบบต้องรองรับได้โดยไม่พัง
*   **Edge Case:** การทดสอบในเงื่อนไขค่าขอบเขตสุดขั้ว หรือข้อมูลแปลกปลอมที่มักพบบั๊กได้บ่อย
*   **Error Case:** การจำลองระบบล้มเหลว (เช่น เครือข่ายหลุด, ฐานข้อมูลขัดข้อง) เพื่อตรวจดูการกู้คืนและความปลอดภัยของข้อมูล

### 2.2 การแบ่งระดับ/เฟสการทดสอบ (UAT vs. Production Grade)
เนื่องด้วยระบบยังเป็นเวอร์ชันใหม่ที่มีความเสถียรต่ำและยังมีบั๊กตั้งค่าระดับ Unit / Master Data สูง ให้ยึดหลักเกณฑ์ดังนี้ในการเขียนหรือรันเทส:
*   **Grade A (UAT Phase):** เน้นเฉพาะ Happy Path และเวิร์กโฟลว์หลักของผู้ใช้งานก่อน เพื่อให้แน่ใจว่าระบบสามารถรันครบลูปการรักษาได้เป็นหลัก
*   **Grade B (Production Readiness Phase):** เป็นเคสประเภท Validation ที่ลึกขึ้น, Edge Cases, และข้อผิดพลาดระดับเครือข่าย ให้แยกไว้ทำในสเตปเตรียมระบบพร้อมสำหรับ Go-Live (ไม่ควรนำมารันสะสมในช่วงแรกเพราะจะส่งผลให้ทีมพัฒนางานล่าช้าเนื่องจากบั๊กปลีกย่อย)

## 📋 4. รูปแบบและแนวทางการเขียนเทสเคส (Test Case Template & Guidelines)

เมื่อต้องเขียนหรือออกแบบเทสเคสใหม่ ให้ใช้โครงสร้างและแนวทางดังต่อไปนี้เสมอ:

### 4.1 เทมเพลตตารางเทสเคส (Test Case Template)
คอลัมน์มาตรฐานที่ต้องใช้ในการแสดงผลทั้งในไฟล์เอกสาร Markdown และการบันทึกลงไฟล์ Excel:
*   **TC No.**: รหัสเทสเคส (เช่น TC001, TC002)
*   **Linear no.**: รหัสการ์ด Linear ที่เกี่ยวข้อง (เช่น NUH-802)
*   **Detail**: หมวดหมู่ย่อยหรือโมดูลย่อยที่กำลังทดสอบ (เช่น Preferred Bed Sync, Bed Reservation Modal)
*   **Test Case Name**: ชื่อหรือคำอธิบายสิ่งที่ต้องการทดสอบ
*   **Precondition**: เงื่อนไขก่อนการทดสอบ (ข้อมูลหรือสถานะเริ่มต้นที่จำเป็น)
*   **Step Test**: ขั้นตอนการทดสอบแบบทีละขั้นตอน (1, 2, 3...)
*   **BDD**: การอธิบายเงื่อนไขในรูปแบบ Given-When-Then
*   **Expected Result**: ผลลัพธ์ที่คาดหวังจากระบบ
*   **Actual Result**: ผลการทดสอบจริง (เว้นว่างไว้สำหรับรอรัน)
*   **Status**: สถานะการรัน (เว้นว่างไว้สำหรับรอรัน)
*   **Remark**: หมายเหตุอื่น ๆ

### 4.2 แนวปฏิบัติในการออกแบบเทสเคส (Design Patterns)
1.  **Black Box & UI Scope:** เน้นทดสอบการใช้งานแบบ Black Box ผ่านอินเตอร์เฟสผู้ใช้ (`cortex-ui`) เป็นหลัก (ยกเว้นผู้ใช้จะระบุให้เขียนเทสระดับ API/Integration)
2.  **Granularity (การแยกเคสย่อย):** ในกรณีที่มีการตรวจสอบฟิลด์หลาย ๆ ฟิลด์ (เช่น การตรวจสอบชื่อจริงแทน Raw ID ตาม AC) แนะนำให้**แยกแยกหัวข้อการทดสอบเป็นรายข้อ (รายฟิลด์)** เพื่อความสะดวกในการติดตามบั๊กและบันทึกผลการทดสอบ แทนการมัดรวมกันเป็นข้อเดียว
3.  **Happy/Unhappy & Fallback:** ออกแบบให้ครอบคลุมทั้งกรณีปกติ (Happy Path), กรณีที่ไม่ระบุข้อมูล (เช่น ไม่ใส่เตียงสำรอง), และกรณีระบบต้องใช้ตัวเลือกรอง (Fallback)

## 📥 5. รูปแบบ Template สำหรับการนำเข้าข้อมูล (Import) และค่า Master Data

เมื่อออกแบบ Scenario หรือ Test Cases สำหรับนำเข้าเข้าระบบ (Import) ผ่านไฟล์ Excel (.xlsx) ให้ปฏิบัติตามโครงสร้างและค่า Master Data ดังนี้:

#### A) Template สำหรับนำเข้า Scenarios (E2E Scenarios)
*   **ชื่อชีตแรก:** `Scenarios Template`
*   **โครงสร้างคอลัมน์ (ตามลำดับ):**
    `Scenario ID` | `Scenario Name` | `Module` | `Case Type` | `Objective` | `Priority` | `Related Risk` | `Card ID`
*   **ข้อกำหนดคอลัมน์หลัก:**
    *   `Scenario ID` (จำเป็น): รหัส Scenario ไม่ซ้ำกัน เช่น TS001, TS002
    *   `Scenario Name` (จำเป็น): ชื่อ/คำอธิบาย Scenario
    *   `Module` (จำเป็น): โมดูลหลัก (ใช้ค่าจากข้อ 5.2.1)
    *   `Card ID` (ไม่จำเป็น): รหัสการ์ดเพื่อผูกตรงๆ เช่น NEX-4546 (ถ้าว่าง ระบบจะสร้าง Staging Card ให้อัตโนมัติ)

#### B) Template สำหรับนำเข้า Test Cases
*   **ชื่อชีตแรก:** `Test Cases Template`
*   **โครงสร้างคอลัมน์ (ตามลำดับ):**
    `Test Case ID` | `Scenario ID` | `Module` | `Feature` | `Case Type` | `Test Case Description` | `Steps` | `Preconditions` | `Test Data` | `Expected Result` | `Priority` | `Severity` | `Impact` | `Test Level` | `Test Type` | `Version`
*   **ข้อกำหนดคอลัมน์หลัก:**
    *   `Test Case ID` (จำเป็น): รหัสเทสเคส เช่น TC001, TC-OPD-001
    *   `Scenario ID` (ไม่จำเป็น): รหัส Scenario ที่จะผูก Traceability
    *   `Module` (จำเป็น): โมดูลหลัก (ใช้ค่าจากข้อ 5.2.1)
    *   `Case Type` (จำเป็น): ประเภทการทดสอบ (ใช้ค่าจากข้อ 5.2.2)
    *   `Test Case Description` (จำเป็น): คำอธิบายกรณีทดสอบ
    *   `Steps` (จำเป็น): ขั้นตอนการทดสอบอย่างละเอียด (แนะนำเลขข้อ 1., 2., 3.)
    *   `Expected Result` (จำเป็น): ผลลัพธ์ที่คาดหวัง

   * ให้ใช้รูปแบบรหัสที่คั่นด้วยเครื่องหมายขีดกลางตามมาตรฐานเดิมของ Master Template เช่น **`TC-NUH-802-001`** (ห้ามเขียนติดกันเป็น `TC-NUH802-001` หรือ `TC001` เด็ดขาด)
4. **การเขียนข้อมูลลงไฟล์ Excel (Smart Append Rules):**
   * **ห้ามเขียนทับสูตรหรือทำลายโครงสร้างเดิมเด็ดขาด:** ก่อนเขียนข้อมูลลง Excel ต้องตรวจสอบว่าคอลัมน์ใดมีสูตร (Formulas) หรือ Data Validation (Dropdowns) อยู่ (เช่น คอลัมน์ `Feature`, `Create Date`, `Review Date` ฯลฯ)
   * **คัดลอกสูตรจากบรรทัดต้นแบบเสมอ:** กรณีที่คอลัมน์มีสูตร ให้ใช้วิธีคัดลอกสูตรจากบรรทัดแรกๆ (Row Template) และปรับแก้เลขบรรทัด (Row Number) ในสูตรให้ตรงกับบรรทัดที่กำลังเพิ่มข้อมูล (หากเป็นสูตรอ้างอิง Table ให้อนุรักษ์โครงสร้างเดิมไว้)
   * **การขยายขอบเขต Table (Table & Validation Expansion):** เมื่อมีการต่อท้ายข้อมูล (Append) นอกขอบเขตของตาราง (เช่น ตาราง `Master_TCM` หรือ `Scenarios`) ต้องทำการแก้ไข Property ของ Excel Table (Table Range) และ Data Validation ให้ครอบคลุมบรรทัดที่เพิ่มเข้ามาใหม่ด้วย
   * **บังคับใส่ Requirement Detail เสมอ:** ทุก Test Case ในชีต `1. Master_TCM` ต้องมีการระบุชื่อการ์ด (Card Name) ในคอลัมน์ `Requirement Detail` ให้ครบถ้วน
   * **ต้องอัปเดตชีต 2. Scenarios ด้วยเสมอ:** ทุกครั้งที่เพิ่มเทสเคสของการ์ดใหม่ ต้องสร้างและนำข้อมูลสรุปไปบันทึกลงชีต `2. Scenarios` ให้สอดคล้องกันด้วย
5. **ข้อควรระวังในการ Export / เขียนข้อมูลผ่าน Script (Data Mapping & Defaults Logic):**
   * **อ้างอิงชื่อคอลัมน์ให้ตรงเป๊ะ 100%:** เมื่อต้องเขียน Script / Code เพื่อวิเคราะห์และบันทึกข้อมูลลง Excel ต้องตรวจสอบชื่อคอลัมน์จริงในไฟล์เสมอ (เช่น `Pre-conditions` มีตัว s และมีขีดกลาง, `Test Steps` เติม s, `Requirment Detail` สะกดตก e เป็นต้น) ห้ามอ้างอิงตามความเคยชิน
   * **บังคับ Inject ค่า Default ควบคุมระบบ:** ห้ามส่งค่าว่าง (Null/Blank) สำหรับคอลัมน์ระบบ หากไม่มีในโครงสร้างข้อมูลต้นทาง ให้ Script ทำการยัดค่ามาตรฐานลงไปให้ทันที ได้แก่ `Test Level` = `System`, `Test Type` = `Functional`, `Priority` = `Medium`, `Execution Status` = `Draft`, `Run in This Cycle?` = `Yes`, `Create By` = ชื่อ Agent หรือผู้ทำระบบ (เช่น `QA Gemini Agent`), และ `Create Date` = วันที่ปัจจุบันเสมอ
